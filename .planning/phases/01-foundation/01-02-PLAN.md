**Step 1: Create src/lib/auth.ts -- Better Auth server configuration.**

This is the core auth configuration file. It must include:

```typescript
import { betterAuth } from "better-auth";
import { drizzleAdapter } from "better-auth/adapters/drizzle";
import { admin, twoFactor } from "better-auth/plugins";
import { db } from "@/lib/db";
import * as schema from "@/lib/db/schema";
import { kvGet, kvSet, kvDelete } from "@/lib/redis";
import { checkAccountLockout, recordFailedAttempt, clearFailedAttempts } from "@/lib/lockout";

export const auth = betterAuth({
  database: drizzleAdapter(db, {
    provider: "pg",
    schema,
  }),
  emailAndPassword: {
    enabled: true,
    minPasswordLength: 8, // D-07: minimum 8 characters
    resetPasswordTokenExpiresIn: 3600, // D-09: 1 hour in seconds
  },
  session: {
    expiresIn: 60 * 60 * 24 * 30, // D-10: 30 days in seconds
    updateAge: 60 * 60 * 24, // Refresh session every 24 hours
    cookieCache: {
      enabled: true,
      maxAge: 60 * 5, // Cache session in cookie for 5 minutes
    },
  },
  user: {
    additionalFields: {
      role: {
        type: "string",
        defaultValue: "customer",
        input: false, // Not settable during registration -- prevents mass assignment
      },
      phone: {
        type: "string",
        required: true, // D-05: phone required at registration
      },
      centralUserId: {
        type: "string",
        required: false,
        input: false, // Set by central API integration, not by users
      },
    },
  },
  plugins: [
    admin(),
    twoFactor(),
  ],
  secondaryStorage: {
    get: async (sessionId) => {
      const data = await kvGet(`session:${sessionId}`);
      return data ? JSON.parse(data) : null;
    },
    set: async (sessionId, data, ttl) => {
      await kvSet(`session:${sessionId}`, JSON.stringify(data), ttl);
    },
    delete: async (sessionId) => {
      await kvDelete(`session:${sessionId}`);
    },
  },
  databaseHooks: {
    user: {
      create: {
        after: async (user) => {
          // Audit log: user registration
          // Will be fully wired in Task 2
        },
      },
    },
    session: {
      create: {
        after: async (session) => {
          // Audit log: user login -- clears lockout on successful login
          // Will be fully wired in Task 2
        },
      },
    },
  },
  hooks: {
    before: [
      {
        matcher: (context) => context.path === "/sign-in/email",
        handler: async (context) => {
          // D-08: Account lockout check before sign-in
          // Extract email from request body
          const body = context.body as Record<string, unknown>;
          const email = body?.email as string;
          if (email) {
            const lockoutStatus = await checkAccountLockout(email);
            if (lockoutStatus.locked) {
              throw new Error(
                `Account temporarily locked. Try again in ${lockoutStatus.remainingMinutes} minutes.`
              );
            }
          }
          return context;
        },
      },
    ],
    after: [
      {
        matcher: (context) => context.path === "/sign-in/email",
        handler: async (context) => {
          const body = context.body as Record<string, unknown>;
          const email = body?.email as string;
          if (email) {
            // D-08: Record failed attempt or clear on success
            const response = context.response as Record<string, unknown>;
            const success = response?.status !== "error" && !(response as Record<string, unknown>)?.error;
            if (success) {
              await clearFailedAttempts(email);
            } else {
              await recordFailedAttempt(email);
            }
          }
          return context;
        },
      },
    ],
  },
});

export type Auth = typeof auth;
```

**Key design decisions:**
- `input: false` on `role` field prevents users from setting their own role during registration (mass assignment protection, D-02/AUTH-02)
- `secondaryStorage` always provided (uses in-memory fallback from Plan 01 when Redis unavailable) per D-14
- Session duration 30 days per D-10
- Min password length 8 per D-07
- `resetPasswordTokenExpiresIn: 3600` per D-09 (1 hour = 3600 seconds)
- `admin()` plugin enables admin user management
- `twoFactor()` plugin enables TOTP 2FA readiness per AUTH-06
- `hooks.before` checks account lockout before sign-in per D-08
- `hooks.after` records failed attempts or clears lockout on success per D-08
- The exact Better Auth hooks API (matcher, handler, context shape) should be verified against Better Auth docs during implementation. If the hooks API differs, the lockout can alternatively be implemented as a wrapper around signIn in auth-client.ts or as a pre-check in the SignInForm component calling a custom API endpoint.

**Step 1: Create src/lib/lockout.ts -- Account lockout module (D-08).**

This module implements D-08: 5 failed login attempts triggers 15-minute account lockout.

Uses Redis kvGet/kvSet/kvDelete for the failed attempt counter (works with in-memory fallback too).

```typescript
import { kvGet, kvSet, kvDelete } from "@/lib/redis";

const MAX_FAILED_ATTEMPTS = 5;
const LOCKOUT_DURATION_SECONDS = 15 * 60; // 15 minutes

interface LockoutStatus {
  locked: boolean;
  remainingMinutes?: number;
  failedAttempts: number;
}

function getLockoutKey(email: string): string {
  return `lockout:${email.toLowerCase()}`;
}

function getAttemptsKey(email: string): string {
  return `lockout:attempts:${email.toLowerCase()}`;
}

/**
 * Check if an account is currently locked out.
 * Returns lockout status including whether locked and remaining time.
 */
export async function checkAccountLockout(email: string): Promise<LockoutStatus> {
  const lockoutKey = getLockoutKey(email);
  const attemptsKey = getAttemptsKey(email);

  const lockoutData = await kvGet(lockoutKey);
  if (lockoutData) {
    const lockoutExpiry = parseInt(lockoutData, 10);
    const remainingMs = lockoutExpiry - Date.now();
    if (remainingMs > 0) {
      return {
        locked: true,
        remainingMinutes: Math.ceil(remainingMs / 60000),
        failedAttempts: MAX_FAILED_ATTEMPTS,
      };
    }
    // Lockout expired, clear it
    await kvDelete(lockoutKey);
  }

  const attemptsData = await kvGet(attemptsKey);
  const failedAttempts = attemptsData ? parseInt(attemptsData, 10) : 0;

  return {
    locked: false,
    failedAttempts,
  };
}

/**
 * Record a failed login attempt.
 * If attempts reach MAX_FAILED_ATTEMPTS, set the lockout.
 */
export async function recordFailedAttempt(email: string): Promise<LockoutStatus> {
  const attemptsKey = getAttemptsKey(email);
  const lockoutKey = getLockoutKey(email);

  const currentData = await kvGet(attemptsKey);
  const currentAttempts = currentData ? parseInt(currentData, 10) : 0;
  const newAttempts = currentAttempts + 1;

  if (newAttempts >= MAX_FAILED_ATTEMPTS) {
    // Lock the account for 15 minutes
    const lockoutExpiry = Date.now() + LOCKOUT_DURATION_SECONDS * 1000;
    await kvSet(lockoutKey, lockoutExpiry.toString(), LOCKOUT_DURATION_SECONDS);
    // Clear the attempts counter since lockout is active
    await kvDelete(attemptsKey);
    return {
      locked: true,
      remainingMinutes: 15,
      failedAttempts: newAttempts,
    };
  }

  // Store incremented attempt count with 1-hour TTL (auto-cleanup if no more attempts)
  await kvSet(attemptsKey, newAttempts.toString(), 3600);
  return {
    locked: false,
    failedAttempts: newAttempts,
  };
}

/**
 * Clear failed attempts on successful login.
 */
export async function clearFailedAttempts(email: string): Promise<void> {
  const attemptsKey = getAttemptsKey(email);
  await kvDelete(attemptsKey);
}
```

**Key design:**
- Lockout key stores the absolute expiry timestamp (epoch ms) so we can compute remaining time
- Attempts key stores the count with a 1-hour TTL for auto-cleanup
- Both keys use lowercase email for case-insensitive matching
- Works with in-memory fallback (no Redis required for dev)
- `checkAccountLockout` is called in the `hooks.before` sign-in hook in auth.ts (configured in Task 1)
- `recordFailedAttempt` is called in the `hooks.after` sign-in hook when login fails
- `clearFailedAttempts` is called in the `hooks.after` sign-in hook when login succeeds

```typescript
import { db } from "@/lib/db";
import { auditLogs } from "@/lib/db/schema";

type AuditAction =
  | "user.registered"
  | "user.login"
  | "user.login_failed"
  | "user.logout"
  | "user.password_reset"
  | "user.email_verified"
  | "user.account_locked"
  | "user.role_changed"
  | "user.banned"
  | "user.activated"
  | "order.created"
  | "order.status_changed"
  | "license.created"
  | "license.status_changed"
  | "ticket.created"
  | "ticket.updated"
  | "admin.setup_completed"
  | string;

interface AuditLogEntry {
  actorId?: string;
  actorRole?: string;
  action: AuditAction;
  targetType?: string;
  targetId?: string;
  details?: Record<string, unknown>;
  ipAddress?: string;
}

export async function createAuditLog(entry: AuditLogEntry): Promise<void> {
  try {
    await db.insert(auditLogs).values({
      actorId: entry.actorId ?? null,
      actorRole: entry.actorRole ?? null,
      action: entry.action,
      targetType: entry.targetType ?? null,
      targetId: entry.targetId ?? null,
      details: entry.details ?? null,
      ipAddress: entry.ipAddress ?? null,
    });
  } catch (error) {
    // Audit logging should never break the main flow
    console.error("[Audit] Failed to create audit log:", error);
  }
}
```

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Client -> /api/auth/* | Untrusted HTTP requests to Better Auth API routes |
| Better Auth -> PostgreSQL | Auth data persistence with parameterized queries |
| Better Auth -> Redis | Session storage via secondaryStorage interface |
| App -> Auth config | Server-side auth configuration, no client exposure |
| Attacker -> Login form | Brute force attacks on login endpoint |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-05 | T (Tampering) | user.role field | mitigate | `input: false` on role additionalField prevents mass assignment during registration. Only server-side code or admin API can change roles. |
| T-01-06 | S (Spoofing) | Session tokens | mitigate | Better Auth generates cryptographically secure session tokens. Stored in HttpOnly cookies. Redis secondaryStorage validates session existence on every request. |
| T-01-07 | E (Elevation) | RBAC bypass | mitigate | admin() plugin provides server-side role checks. Role checks happen in page components (server-side), not just in proxy. |
| T-01-08 | I (Information Disclosure) | Audit logs | accept | Audit logs contain actor ID, action, target -- no passwords or sensitive tokens. Access restricted to admin routes. |
| T-01-09 | D (Denial of Service) | Redis unavailable | mitigate | secondaryStorage uses kvGet/kvSet/kvDelete which gracefully fall back to in-memory store when Redis is down. |
| T-01-09b | D (Denial of Service) | Brute force login | mitigate | Account lockout after 5 failed attempts for 15 minutes (D-08). Lockout counter in Redis with TTL auto-cleanup. Prevents credential stuffing attacks. |
</threat_model>
