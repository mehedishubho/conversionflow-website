<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| User -> Auth forms | Untrusted input on login/register/reset forms |
| Auth forms -> Better Auth API | Client-side API calls to /api/auth/* |
| Better Auth -> Resend API | Email delivery via third-party API |
| User -> Email link | Password reset and verification links in email |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-10 | T (Tampering) | Login form | mitigate | Better Auth handles credential validation server-side. Account lockout after 5 failures (D-08) implemented in Plan 02 lockout.ts module. SignInForm displays lockout error messages. |
| T-01-11 | S (Spoofing) | Password reset link | mitigate | Reset token is single-use, expires in 1 hour per D-09 (configured in auth.ts resetPasswordTokenExpiresIn: 3600). Better Auth generates cryptographically secure tokens. |
| T-01-12 | T (Tampering) | Registration form | mitigate | Server-side validation via Better Auth. Role field has `input: false` preventing mass assignment. minPasswordLength enforced by Better Auth. |
| T-01-13 | I (Information Disclosure) | Email content | accept | Verification/reset emails contain only the user's email and a link. No sensitive data exposed beyond what the user already knows. |
| T-01-14 | D (Denial of Service) | Resend API | accept | Email sending failures should not block registration (D-06: auto-login regardless). Log errors but continue. |
</threat_model>
