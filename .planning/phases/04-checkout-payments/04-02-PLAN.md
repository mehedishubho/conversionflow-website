Create 5 API route handlers under `src/app/api/ssl-commerz/`:

**1. `create-session/route.ts` (POST)** -- per D-12:
- Authenticate user via `auth.api.getSession({ headers: await headers() })`
- Parse JSON body: { plan, couponCode, discountAmount, taxAmount, totalAmount }
- Validate plan against pricingTiers -- compute authoritative price server-side
- Create a pending order in DB first (to get the order ID as tranId)
- Call `createSSLSession()` with:
  - tranId: order.id (UUID)
  - totalAmount: server-computed total (NOT client value)
  - successUrl: `${APP_URL}/api/ssl-commerz/success`
  - failUrl: `${APP_URL}/api/ssl-commerz/fail`
  - cancelUrl: `${APP_URL}/api/ssl-commerz/cancel`
  - ipnUrl: `${APP_URL}/api/ssl-commerz/ipn`
  - Customer info from session.user (name, email, phone)
  - Custom values: value_a=order.id, value_b=userId, value_c=plan, value_d=couponCode
- Return `{ url: GatewayPageURL, orderId: order.id }` to client for redirect

**2. `success/route.ts` (GET)** -- per D-12:
- Read query params: val_id, tran_id (order ID), status
- Redirect customer to `/dashboard/checkout/success?order=${tran_id}&status=completed`
- DO NOT mark order completed here -- that's IPN's job (per Pitfall 3)

**3. `fail/route.ts` (GET)**:
- Redirect to `/dashboard/checkout/success?order=${tran_id}&status=failed`

**4. `cancel/route.ts` (GET)**:
- Redirect to `/dashboard/checkout?plan=X` (extract plan from query params or session)

**5. `ipn/route.ts` (POST)** -- per D-12, D-13, Pitfall 3 (CRITICAL):
- Read form data: val_id, tran_id (our order UUID), status, bank_tran_id, amount, currency
- Call `validateSSLPayment(val_id)` for server-to-server validation
- If validation status !== "VALID": return 400
- Find order by tran_id in DB. If not found: return 404
- If order already completed: return 200 (idempotent)
- Update order: set status "completed", paymentRef to bank_tran_id
- Create audit log: `createAuditLog({ action: "order.status_changed", targetType: "order", targetId: order.id, details: { from: "pending", to: "completed", paymentRef: bank_tran_id } })`
- Call `importOrderToCentral()` (or mock if no API key). Payload includes order data + user data (query user by order.userId)
- If central API succeeds:
  - Update order.centralOrderId with response.centralOrderId
  - Update user.centralUserId with response.centralUserId
  - Insert into licenses table: userId, orderId, productId, plan, licenseKey from response, status "active", centralLicenseId from response
  - Create audit log: `license.created`
- If central API fails:
  - Order stays "completed" but centralOrderId is null -- this is the "pending_sync" state (D-14)
  - Log the failure for admin visibility
- Return 200 JSON `{ ok: true }`

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| SSL Commerce -> IPN handler | External callback: must validate via server-to-server call, never trust redirect data |
| client -> create-session | Untrusted: plan, couponCode, amounts from client form |
| SSL Commerce API | External service: HTTPS required, credentials in env vars |
| Central Licensing API | External service: Bearer token auth, HTTPS required |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-06 | Tampering | IPN handler | mitigate | Validate via server-to-server call to validateSSLPayment(val_id). Never trust redirect/form data alone. Check validation.status === "VALID". |
| T-04-07 | Tampering | create-session | mitigate | Compute final amount server-side from pricingTiers data. Never use client-submitted amount for SSL session. |
| T-04-08 | Spoofing | IPN handler | mitigate | SSL Commerz val_id is single-use and verified server-to-server. Reject if validation fails. |
| T-04-09 | Repudiation | IPN handler | mitigate | Audit log all order status changes with actor, action, target, and details. |
| T-04-10 | Information Disclosure | Central API calls | accept | Order data sent to central API is internal. HTTPS protects in transit. Bearer token auth prevents unauthorized access. |
| T-04-11 | Denial of Service | IPN handler | accept | SSL Commerz retries IPN on failure. Handler is idempotent (checks if order already completed). |
</threat_model>
