Create `src/app/(portal)/actions/checkout.ts` with `"use server"` directive. Export three server actions:

**1. `validateCoupon(code: string, orderAmount: number)` -- per D-06:**
- Auth check: `auth.api.getSession({ headers: await headers() })`, redirect if no session
- Use `db.transaction()` for race-condition safety (Pitfall 2 from RESEARCH)
- Query coupons table where `code` matches AND `active` is true
- Validate: not expired (expiresAt check), usage limit not reached (currentUses < maxUses when maxUses is not null), minimum order amount met
- Calculate discount: if type is "percentage" -> `Math.round(orderAmount * coupon.value / 100)`, capped at orderAmount; if type is "flat" -> `Math.min(coupon.value, orderAmount)`
- Increment `currentUses` by 1 within the same transaction (this reserves the coupon usage)
- Return `{ success: true, discount, type, value }` or `{ error: string }`
- Error messages per UI-SPEC: "Invalid coupon code", "This coupon has expired", "This coupon has reached its usage limit", "Minimum order amount for this coupon is {amount} BDT"

**2. `createManualOrder(formData: FormData)` -- per D-07, D-08, D-21:**
- Auth check with session
- Extract from formData: plan, paymentMethod, paymentRef (transaction ID), couponCode, amount, taxAmount, discountAmount
- Validate plan against `pricingTiers` from `@/data/pricing.ts` -- compute base price from plan, NEVER trust client amount
- Validate paymentMethod against `paymentMethodEnum` enumValues
- For manual methods (bkash, nagad, rocket, bank_transfer): paymentRef is REQUIRED. Validate min length 4 chars.
- **Dedup check (D-21):** Query orders where `paymentRef` matches the submitted value. If found, return error "This transaction ID has already been used."
- Generate display order ID using nanoid: `"CF-" + nanoid(8)`. Store the UUID as `id` and use nanoid as... actually the schema has `id` as UUID PK. Use the UUID as the primary key but generate a display-friendly ID. Since the schema doesn't have a displayId column, just use `id.slice(0, 8)` for display (established pattern from InvoiceTable).
- Insert into orders table with status "pending"
- Create audit log: `createAuditLog({ action: "order.created", targetType: "order", targetId: order.id })`
- Return `{ success: true, orderId: order.id }`

**3. `calculateVAT(amount: number)` -- per D-15, D-16:**
- Query settings table for `vat_rate` and `vat_mode`
- Default: rate 15, mode "exclusive" if settings not found
- If mode is "exclusive": taxAmount = `Math.round(amount * rate / 100)`, total = amount + taxAmount
- If mode is "inclusive": taxAmount = `Math.round(amount * rate / (100 + rate))`, total = amount
- Return `{ taxAmount, total, rate, mode }`

**4. `getPaymentAccounts()` -- per D-10:**
- Query paymentAccounts where `active` is true, group by method
- Return Record<paymentMethod, { accountName, accountNumber, bankName, branch, routingNumber, instructions }>
- No auth required (used on checkout page to display payment instructions)

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client -> createManualOrder | Untrusted: plan, paymentMethod, paymentRef, amounts sent from client form |
| client -> validateCoupon | Untrusted: coupon code string from client |
| DB -> coupon validation | Race condition risk on currentUses increment |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-01 | Tampering | createManualOrder | mitigate | Server-side price computation from pricingTiers data, never trust client amount. Verify plan exists in pricingTiers and use the authoritative BDT integer price. |
| T-04-02 | Tampering | createManualOrder | mitigate | Dedup check on paymentRef column (D-21). Query existing orders before insert. |
| T-04-03 | Tampering | validateCoupon | mitigate | Transactional validation + increment within db.transaction(). Prevents race condition on currentUses exceeding maxUses. |
| T-04-04 | Information Disclosure | getPaymentAccounts | accept | Payment account details (numbers, instructions) are public-facing on checkout page. No sensitive data exposed beyond what customer needs to make payment. |
| T-04-05 | Elevation of Privilege | createManualOrder | mitigate | Auth check via auth.api.getSession. Redirect to /login if no session. User can only create orders for themselves (userId from session). |
</threat_model>
