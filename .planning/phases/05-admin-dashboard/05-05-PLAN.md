    **1. Create `src/app/(admin)/actions/admin-notifications.ts`** (per D-22, D-23):

    - `"use server"` directive.
    - Import `db` from `@/lib/db`, `notifications` from schema, `eq, and, desc, sql, inArray` from drizzle-orm.
    - Import `auth` from `@/lib/auth`, `headers` from `next/headers`, `redirect` from `next/navigation`.

    **Admin notification types (per D-23):**
    - Define constant `ADMIN_NOTIFICATION_TYPES = ["payment_failed", "license_expiring", "new_signup", "new_ticket", "fraud_alert"]`.

    **`getAdminNotifications()` server action:**
    - Get session via `auth.api.getSession({ headers: await headers() })`.
    - If no session, return `{ notifications: [], unreadCount: 0 }`.
    - Get user role: `(session.user as Record<string, unknown>).role as string`.
    - If role is not "admin" or "super_admin", return empty (non-admin users don't get admin notifications).
    - Query notifications where `type IN ADMIN_NOTIFICATION_TYPES` and `userId = session.user.id`, ordered by `createdAt DESC`, limit 10.
    - Query unread count where same conditions plus `read = false`.
    - Return `{ notifications, unreadCount }`.

    **`markAdminNotificationRead(notificationId: string)` server action:**
    - Session check. Verify notification belongs to current user AND type is in ADMIN_NOTIFICATION_TYPES.
    - Update `read = true`.
    - Return `{ success: true }`.

    **`markAllAdminNotificationsRead()` server action:**
    - Session check + admin role check.
    - Update all notifications where `userId = session.user.id` AND `type IN ADMIN_NOTIFICATION_TYPES` AND `read = false` to `read = true`.
    - Return `{ success: true }`.

    **`createAdminNotification(params: { adminUserId: string, type: string, title: string, message: string, data?: Record<string, unknown> })` server action:**
    - This is a utility function called by other server actions when admin-relevant events occur.
    - Validate type is in ADMIN_NOTIFICATION_TYPES.
    - Insert into notifications table: `{ userId: adminUserId, type, title, message, data, read: false }`.
    - Return `{ success: true }`.
    - NOTE: This function is called proactively by existing server actions. For now, only define the function. Wiring it into existing actions (verifyOrder failure, user registration, ticket creation) is optional and can happen during execution if time permits.
